# 后端开发一：客户与联系人模块需求文档

## 一、开发身份

本人负责课程设计中的 **后端开发一：客户与联系人模块**。

根据任务书要求，本角色负责：

- 客户 CRUD
- 联系人 CRUD
- 客户与联系人相关业务规则
- 后端数据验证
- `CustomerController`
- `ContactController`
- 与前端 Razor 页面或 Ajax 接口对接

项目采用 DDD 分层设计，目录结构如下：

```text
CRM.Application
CRM.Application.Contracts
CRM.Domain
CRM.Domain.Shared
CRM.Infrastructure
CRM.Web
```

## 二、开发范围

本次开发只负责 **客户管理模块** 和 **联系人管理模块**。

不负责：

- 合同管理
- 合同明细
- 回款计划
- 数据字典维护
- Git 仓库管理
- 整体 UI 美化

但客户详情页需要预留展示客户下合同列表的位置，具体合同数据可由后端开发二对接。

## 三、业务需求

### 1. 客户管理

客户字段包括：

- 客户 Id
- 企业名称，必填，唯一
- 统一社会信用代码
- 所属行业
- 省
- 市
- 区
- 详细地址
- 备注
- 创建时间
- 是否删除 / 是否禁用

需要实现功能：

- 客户列表
- 按企业名称、行业、创建时间查询
- 分页查询，默认每页 10-20 条
- 新增客户
- 编辑客户
- 删除客户
- 客户详情

业务规则：

- 企业名称不能为空。
- 企业名称不能重复。
- 新增和编辑时需要去除首尾空格。
- 如果客户名下已经存在合同，禁止物理删除，只能逻辑删除或禁用。
- 被删除或禁用的客户不能继续新增联系人。
- 客户详情页需要展示客户基本信息、联系人列表、合同列表。

### 2. 联系人管理

联系人字段包括：

- 联系人 Id
- 所属客户 Id
- 姓名，必填
- 职务
- 手机号
- 邮箱
- 是否关键决策人

需要实现功能：

- 在客户详情页查看联系人列表
- 按姓名、职务筛选联系人
- 新增联系人
- 编辑联系人
- 删除联系人

业务规则：

- 联系人不能脱离客户独立存在。
- 新增联系人时必须指定客户。
- 联系人姓名不能为空。
- 联系人手机号、邮箱需要做基础格式校验。
- 同一客户下是否允许联系人重名：
  - 建议不允许。
  - 如果已有同名联系人，新增时返回业务异常。
- 联系人可以物理删除。
- 如果客户已经被逻辑删除或禁用，不能新增或编辑联系人。

## 四、分层开发要求

### 1. `CRM.Domain`：领域层

领域层负责写核心业务规则。

需要检查或补充：

```text
CRM.Domain/Customers/Customer.cs
CRM.Domain/Customers/Contact.cs
```

`Customer` 中建议包含的方法：

```csharp
UpdateInfo(...)
AddContact(...)
UpdateContact(...)
RemoveContact(...)
MarkAsDeleted(...)
CanDelete(...)
```

应放在领域层的规则：

- 客户名称不能为空。
- 客户信息更新时做 trim。
- 已删除客户不能维护联系人。
- 联系人必须归属于客户。
- 联系人姓名不能为空。
- 同一客户下联系人不能重名。
- 联系人更新时做 trim。
- 逻辑删除客户。

不应放在领域层的内容：

- EF Core 查询
- Controller
- DTO
- 页面逻辑
- 分页查询
- 全库唯一性查询

### 2. `CRM.Application.Contracts`：应用抽象层

这里放应用服务接口和 DTO。

建议补充或完善：

```text
CRM.Application.Contracts/Customers/ICustomerAppService.cs
CRM.Application.Contracts/Customers/Dtos/CustomerDto.cs
CRM.Application.Contracts/Customers/Dtos/CustomerDetailDto.cs
CRM.Application.Contracts/Customers/Dtos/CreateCustomerDto.cs
CRM.Application.Contracts/Customers/Dtos/UpdateCustomerDto.cs
CRM.Application.Contracts/Customers/Dtos/CustomerQueryDto.cs

CRM.Application.Contracts/Contacts/IContactAppService.cs
CRM.Application.Contracts/Contacts/Dtos/ContactDto.cs
CRM.Application.Contracts/Contacts/Dtos/CreateContactDto.cs
CRM.Application.Contracts/Contacts/Dtos/UpdateContactDto.cs
CRM.Application.Contracts/Contacts/Dtos/ContactQueryDto.cs
```

`ICustomerAppService` 建议方法：

```csharp
Task<PagedResultDto<CustomerDto>> GetListAsync(CustomerQueryDto input);
Task<CustomerDetailDto> GetDetailAsync(int id);
Task<CustomerDto> GetByIdAsync(int id);
Task CreateAsync(CreateCustomerDto input);
Task UpdateAsync(UpdateCustomerDto input);
Task DeleteAsync(int id);
```

`IContactAppService` 建议方法：

```csharp
Task<List<ContactDto>> GetListAsync(ContactQueryDto input);
Task<ContactDto> GetByIdAsync(int id);
Task CreateAsync(CreateContactDto input);
Task UpdateAsync(UpdateContactDto input);
Task DeleteAsync(int id);
```

DTO 中使用 `DataAnnotations` 做后端表单验证，例如：

- `[Required]`
- `[MaxLength]`
- `[EmailAddress]`
- `[Phone]`

### 3. `CRM.Application`：应用层

这里实现应用服务，不放 Controller。

建议文件：

```text
CRM.Application/Customers/CustomerAppService.cs
CRM.Application/Contacts/ContactAppService.cs
```

`CustomerAppService` 负责：

- 查询客户列表
- 客户分页
- 客户详情
- 新增客户
- 编辑客户
- 删除客户
- 检查客户名称唯一
- 删除客户前检查是否有关联合同
- DTO 与领域对象转换
- 调用仓储保存数据

`ContactAppService` 负责：

- 查询某客户下联系人
- 新增联系人
- 编辑联系人
- 删除联系人
- 检查客户是否存在
- 调用 `Customer.AddContact()`、`Customer.RemoveContact()` 等领域方法
- DTO 与领域对象转换
- 调用仓储保存数据

应用层可以调用领域层业务方法，但不要把所有业务规则都写成普通 if 堆在应用服务里。

### 4. `CRM.Web`：Web 层

这里放 MVC Controller。

建议文件：

```text
CRM.Web/Controllers/CustomerController.cs
CRM.Web/Controllers/ContactController.cs
```

`CustomerController` 负责：

- 客户列表页面
- 客户详情页面
- 新增客户页面
- 编辑客户页面
- 删除客户接口
- 调用 `ICustomerAppService`

`ContactController` 负责：

- 联系人新增接口
- 联系人编辑接口
- 联系人删除接口
- 联系人列表接口
- 调用 `IContactAppService`

Controller 不直接操作 EF Core，也不要直接写复杂业务规则。

### 5. `CRM.Infrastructure`：基础设施层

基础设施层已经完成时，尽量不要改。

只有以下情况才需要改：

- 现有仓储无法分页查询
- 无法按条件查询客户
- 无法 Include 联系人
- 无法判断客户是否有关联合同

如果必须修改，应只扩展通用仓储或增加查询方法，不要在基础设施层写业务规则。

### 6. `CRM.Domain.Shared`：领域共享层

一般不需要改。

可使用其中已有的：

- `BusinessException`
- 通用枚举
- 共享常量

如果需要新增通用异常或共享枚举，需要和小组成员确认，避免影响其他模块。

## 五、必须实现的后端规则

最终代码至少要体现这些规则：

- 客户名称不能为空。
- 客户名称唯一。
- 联系人姓名不能为空。
- 联系人必须属于客户。
- 客户被删除后不能新增联系人。
- 客户有关联合同时不能物理删除。
- 客户删除优先使用逻辑删除。
- 联系人可以物理删除。
- 列表查询支持筛选和分页。
- Controller 只调用应用服务，不直接操作数据库。
- 应用服务调用领域模型方法完成核心业务行为。
