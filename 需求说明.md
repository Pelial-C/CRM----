# 企业客户与合同管理系统 (CRM Lite) - 需求说明书

**文档版本**：V1.0
**编写角色**：项目经理 (PM)
**更新日期**：2026-06-22

## 一、 项目概述

### 1.1 项目背景
某制造企业目前依赖 Excel 台账来管理客户信息、联系人、合同及回款计划。随着业务规模扩大，Excel 管理暴露出数据一致性差、多人协同冲突、历史数据追溯困难、关联查询效率低等痛点。
为解决上述问题，拟开发“企业客户与合同管理系统 (CRM Lite)”，以实现业务流程的数字化和规范化。

### 1.2 系统定位
*   **系统类型**：企业内部业务运营系统 (LOB - Line of Business)。
*   **核心目标**：替代原有 Excel 台账，提高数据一致性、查询效率和业务协同能力。
*   **边界界定**：聚焦于客户资产与合同履约管理，**不涉及**复杂的 RBAC 细粒度权限控制、不涉及外部支付网关对接、不涉及复杂的审批工作流。

## 二、 核心业务流程分析

系统的核心业务流转围绕“客户获取 -> 合同签订 -> 资金回款”展开，主干流程如下：

1.  **客户建档**：业务员录入企业客户基本信息（如企业名称、行业、统一社会信用代码等）。
2.  **联系人维护**：在已建档的客户名下，添加关键联系人（如决策人、财务对接人），并记录联系方式。
3.  **合同签订**：与客户达成合作后，创建合同记录，选择关联的客户和主要联系人，并录入合同总金额、签订日期及合同明细（产品/服务项）。
4.  **回款计划制定**：根据合同条款，在合同下制定分期回款计划（如：预付款 30%，验收款 60%，质保金 10%）。
5.  **回款执行与核销**：业务员根据实际到账情况，更新回款计划的状态（未回款 -> 部分回款 -> 已回款），系统自动计算合同的整体回款进度。

## 三、 功能需求说明

根据业务模块划分，系统主要包含以下功能需求：

### 3.1 客户管理模块 (Customer Management)
*   **客户列表**：支持按企业名称、行业、创建时间进行分页查询和模糊搜索。
*   **新增客户**：录入企业名称（必填、唯一）、统一社会信用代码、所属行业、企业地址（省市区+详细地址）、备注。
*   **编辑/删除客户**：修改客户信息；若客户名下已存在关联合同，则**禁止物理删除**，仅支持逻辑删除或禁用。
*   **客户详情**：查看客户基本信息，并展示该客户名下的所有联系人列表和合同列表。

### 3.2 联系人管理模块 (Contact Management)
*   *业务约束：联系人不能脱离客户独立存在，必须在客户上下文内进行管理。*
*   **联系人列表**：在客户详情页内展示，支持按姓名、职务筛选。
*   **新增联系人**：录入姓名（必填）、职务、手机号、邮箱、是否为关键决策人（布尔值）。
*   **编辑/删除联系人**：修改联系人信息，支持直接删除（物理删除）。

### 3.3 合同管理模块 (Contract Management)
*   **合同列表**：支持按合同编号、客户名称、合同状态（执行中、已完结、已作废）进行查询。
*   **新增合同**：
    *   选择关联客户（下拉框）。
    *   选择该客户下的主要联系人（联动下拉框，根据选择的客户动态加载）。
    *   录入合同编号（系统自动生成或手动输入，需唯一）、合同名称、签订日期、合同总金额。
    *   添加合同明细（产品/服务名称、数量、单价、小计）。
*   **合同详情**：展示合同基本信息、关联客户/联系人信息、合同明细列表以及回款计划列表。

### 3.4 回款计划模块 (Payment Plan Management)
*   *业务约束：回款计划依附于合同存在。*
*   **制定计划**：在合同详情中新增回款期数，录入计划回款日期、计划回款金额、期数说明。
*   **更新回款状态**：
    *   录入实际回款金额和实际回款日期。
    *   系统自动校验：累计实际回款金额不能大于合同总金额。
    *   系统自动计算并更新状态（未回款/部分回款/已结清）。

### 3.5 基础数据模块 (Foundation)
*   **数据字典**：维护系统所需的基础下拉选项，如“行业分类”、“合同状态”、“联系人职务”等（初期可硬编码或简单建表，后期可扩展）。

## 四、 非功能需求

### 4.1 数据一致性与完整性
*   必须保证客户与联系人、合同与明细之间的强一致性（依托 DDD 聚合根和数据库外键约束）。
*   关键业务数据（如合同金额、回款金额）必须保证精度，防止浮点数计算误差。

### 4.2 性能与响应
*   列表查询接口响应时间 < 1秒。
*   支持合理的数据分页（默认每页 10-20 条），避免一次性加载过多数据导致内存溢出。

### 4.3 易用性与前端交互
*   采用 Bootstrap 构建响应式界面，保证在主流浏览器（Chrome, Edge）下的显示效果。
*   表单提交需提供前端（JS/jQuery）与后端（Data Annotations/FluentValidation）双重校验。
*   关键操作（如删除客户、作废合同）需提供二次确认弹窗。

## 五、 角色与权限说明

虽然系统不涉及复杂的权限控制，但为满足基本的数据隔离和操作审计，设定以下两个基础角色：

1.  **系统管理员 (Admin)**：
    *   拥有所有模块的最高操作权限。
    *   负责基础数据（数据字典）的维护。
2.  **普通业务员 (Sales)**：
    *   负责客户、联系人、合同的日常 CRUD 操作。
    *   只能查看和编辑自己创建的数据（*注：若8天内实现数据权限隔离有困难，可先降级为所有业务员共享数据，仅在UI上显示“创建人”字段*）。
